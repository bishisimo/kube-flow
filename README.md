# Kube-Flow

桌面端 **Kubernetes** 管理工具：**连集群**做浏览与运维；需要时用 **编排中心** 管 YAML 资产（清单、版本、跨环境）。

下面是痛点对照与按模块的能力列表。

---

## 痛点速查：你能得到什么应对

### 环境与连接

| 你可能遇到的痛点 | Kube-Flow 怎么应对 |
|------------------|---------------------|
| 本机要切多套集群（开发 / 测试 / 预发），`kubectl` context 换来换去容易搞混 | **多环境**：为每个 context 起好记的名字，配默认命名空间、标签、收藏与排序，在界面里切换当前操作环境，而不是只靠终端里的 `current-context`。 |
| 集群在内网，只能 SSH 跳板进去才有 kubeconfig | **SSH 隧道**：沿用你本机已有的 `~/.ssh/config`，建好隧道再访问 API；连接阶段在界面可见，便于判断是网络、认证还是隧道没起来。 |
| 只想用熟悉的 kubeconfig，不希望工具另搞一套「云账号」 | 环境来自本机 kubeconfig 或经隧道拉取的远程配置；工具管的是「怎么连、怎么记」，不改变你在集群侧的权限模型。 |

### 日常运维与排障（工作台）

| 你可能遇到的痛点 | Kube-Flow 怎么应对 |
|------------------|---------------------|
| Pod 起不来、服务不通，要在多个页面或命令之间查 Deployment / Pod / Event | **工作台**按命名空间浏览工作负载，结合事件、**流式日志**、容器 **Shell**，并在资源之间跳转（例如 Deployment → ReplicaSet → Pod），减少反复 `get` / `describe`。 |
| ConfigMap / Secret 要改，不想先导出到本地文件再 `apply` | 在界面里直接查看与编辑键值，改完再走集群侧更新流程。 |
| 同一个 Deployment 下多个 Pod，日志要一个个 `kubectl logs` | 支持工作负载维度的多 Pod 日志汇总视图（具体以界面为准）。 |
| 改镜像、调 YAML、应用配置之后才发现不对，想有一份「改之前」可对，而不是对着集群里已经变掉的状态干瞪眼 | 在资源详情里用**快照**把关键操作留痕：需要时能翻出来对照、或当作回滚时的参考，减轻「手滑一次就失忆」的焦虑。 |

### 终端中心

| 你可能遇到的痛点 | Kube-Flow 怎么应对 |
|------------------|---------------------|
| 同时开了多个 Pod Shell、又连了节点或跳板，会话散落在各处，切几次就记不清哪个连的是谁 | **终端中心**集中管理会话，按环境分组，并用「Pod / 主机」区分类型，减少在多个页面里找终端的脑力负担。 |
| 只想先 SSH 到节点或跳板敲几条命令，不想为了进终端必须先在工作台把环境「打开」一遍 | 可直接选定环境**打开主机终端**，不依赖该环境是否已出现在工作台左侧「已打开」列表里。 |

### 日志中心

| 你可能遇到的痛点 | Kube-Flow 怎么应对 |
|------------------|---------------------|
| 排障时要轮流看多个 Pod 或 Workload 的日志，每次从资源列表点进去、离开就像断了线 | 从工作台进入后，**日志会话**留在日志中心，侧栏按环境列出，在多条会话之间切换，减少反复搜索资源、重开面板的次数。 |
| 跟一条日志跟很久，一离开当前资源页就像丢了上下文，又要重新找 Pod、重新开流 | 会话在侧栏按环境挂着，持续跟进时回到日志中心即可续看，不必在资源列表里反复定位同一个对象。 |
| 既要看应用输出，又要看 kube-flow 自身后台在干什么，混在一起没法查 | 同一入口下分 **资源日志** 与 **调试日志**，各看各的，避免应用排障与客户端自证搅在一屏。 |

### 配置资产与发布节奏（编排中心）

| 你可能遇到的痛点 | Kube-Flow 怎么应对 |
|------------------|---------------------|
| YAML 散落在仓库、笔记本和集群里，不知道「应当以哪一份为准」 | **编排中心**在**每个环境下**单独维护一份 **YAML 资产**：保存、对比、再应用到集群；资源可按**组件**归类，方便按业务模块管理。 |
| 一次改了很多对象，担心 `kubectl apply -f` 顺序不对或漏资源 | 编排中心对应用顺序有约定（按资源类型排序），并支持**组件级批量应用**，减少手工排序和漏发。 |
| 在工作台上临时改好了 Deployment / Service 等，希望纳入长期维护的「清单」 | 支持从**工作台**把当前资源 **「同步到编排中心」**，把线上/当前集群状态拉回编排资产里统一改、统一发。 |
| 同一套应用在测试环境验证后，要搬到预发、生产，希望有可追踪的「包」而不是复制粘贴 YAML | **应用包**支持版本化，并可将某版本**同步到其他环境**的编排资产，便于复用与发布节奏管理（具体能力以当前版本界面为准）。 |
| 担心编排中心「下发」覆盖策略和平时在工作台改资源不一致 | 在**设置**里可单独配置 **「编排中心下发」**策略；说明里会区分：编排中心投放规则 vs 工作台编辑现有资源的逻辑。 |
| 编排资产里的 YAML 已经改了好几轮，发版前想回到「某次保存 / 应用」时的那一版，而不是在聊天和文件里翻谁覆盖谁 | **历史快照**按时间留下每次保存与应用的痕迹，挑一版回到编辑区再发，把「回溯」变成可点选的动作，而不是靠记忆。 |

## 功能一览（按模块）

- **环境管理**
  - 本地 kubeconfig 环境管理
  - 基于 ssh config 的 SSH 环境管理

- **工作台**
  - 多环境打开与切换
  - 按命名空间浏览与运维资源
  - 事件
  - 流式日志
  - 容器 Shell
  - ConfigMap / Secret 查看与编辑
  - 资源关联跳转
  - 变更快照

- **终端中心**
  - Pod Shell 与主机 SSH 会话集中管理
  - **主机 SSH 终端**：不必先在工作台「打开」该环境即可连接

- **日志中心**
  - Pod / Workload 多路日志会话
  - 资源日志
  - 调试日志

- **编排中心**
  - 按环境维护 YAML 资产与组件分组
  - 单资源保存、对比、应用
  - 组件级批量应用
  - 保存与应用历史快照
  - 工作台同步至编排资产
  - 应用包版本与跨环境同步

## 安装

### 1. 构建环境

- [Bun](https://bun.sh)：安装依赖与运行脚本
- [Rust](https://www.rust-lang.org/tools/install)：`cargo`，供 Tauri 编译

完整环境说明见 [Tauri 前置条件](https://v2.tauri.app/start/prerequisites/)。

> **注（Windows）**：WebView2 可预先下载并放到自定义目录，在 `bundle.windows.webviewInstallMode` 中指定；见 [Tauri · Windows 安装包](https://v2.tauri.app/distribute/windows-installer/)。

### 2. 构建

在仓库根目录执行：

```bash
bun install
bun run tauri build
```

`tauri build` 会先执行前端的 `bun run build`，再编译 Rust 并生成安装包。

### 3. 安装（按系统）

产物位于 `src-tauri/target/release/bundle/`（具体子目录随目标而变）：

| 系统 | 常见产物 | 安装方式 |
|------|----------|----------|
| macOS | `.dmg`、`.app` | 打开 `.dmg` 将应用拖入「应用程序」 |
| Windows | `.msi`、`.exe` | 运行安装向导 |
| Linux | `.deb`、`.AppImage` 等 | 包管理器安装 `.deb`，或为 `AppImage` 赋予执行权限后运行 |

### 开发调试（可选）

```bash
bun install
bun run tauri dev
```

## 常见问题

**Q：** 如何为环境相关凭证配置安全保护？  
**A：** 打开 **设置 → 安全与凭证**：初始化 **Stronghold** 并设置**主密码**，本地加密保存 SSH 等敏感信息；也可改用系统凭证管理器（无需主密码，由系统保管）。
